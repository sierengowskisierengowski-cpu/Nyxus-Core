Pre-staging more into airootfs/ is worth it, but keep the live ISO “minimal stateful” and push most customization into /etc/skel (for user UX) + systemd units (for behavior). Here are the concrete recommendations for the two questions.

1) Should airootfs/etc/nyxus/ ship with anything?
Yes, but only safe, non-secret defaults and things that help debug/support.

Good candidates to bake in:

/etc/nyxus/nyxus.conf (or .toml/.yaml): defaults like log level, feature toggles, URLs, timeouts.
/etc/nyxus/build-info.conf (or JSON): build metadata so you can identify exactly what ISO someone booted:
git repo + commit SHA
build date/time
archiso version
kernel version
profile version string
/etc/nyxus/firstboot.d/ (optional): drop-in scripts or flags used by a first-boot service.
Do not ship secrets (API keys, private endpoints requiring tokens, user passwords).
I would not ship a “brand lock file” unless you really need anti-remix behavior; it tends to create more pain than value and can break forks/testing.

2) Should .desktop entry + Inter font be baked in?
Bake them in if you want the live session to feel complete without running install.sh.

.desktop entry
If you want “Nyxus Intel” to show up in menus immediately:

put a .desktop file in the image (either system-wide or per-user)
Recommended:

System-wide: airootfs/usr/share/applications/nyxus-intel.desktop
Or per-user default: airootfs/etc/skel/.local/share/applications/nyxus-intel.desktop
If it needs Wayland env vars (Hyprland), include them in Exec= or wrap them in the launcher script.

Inter font
If you want consistent UI typography in the live session:

include the font package via pacman if available (ttf-inter on Arch), rather than copying font files manually.
If you must vendor it, place in airootfs/usr/local/share/fonts/ and run fc-cache at build time (or let the live boot handle it, but that’s slower / less deterministic).
Since you already have a lot of Nerd Fonts listed, adding ttf-inter as a package is usually the cleanest.

Additional “worth it” pre-staging (high impact, low risk)
A) A first-boot service (live-only)
If nyxus-intel needs something to happen automatically on live boot (migrations, generating keys, enabling a user service), add:

a systemd unit in airootfs/etc/systemd/system/
enable it via symlink in airootfs/etc/systemd/system/multi-user.target.wants/
Keep it idempotent and fast.

B) Make failures debuggable
Add a persistent-ish log target (even just good defaults):

ensure journalctl is usable in live mode
optionally set Storage=volatile explicitly (so expectations are clear)
consider a /etc/issue or MOTD line pointing to “collect logs with …”
C) Ensure nyxus-intel dependencies are fully present
Since you moved to /opt/nyxus-intel/*.py, check you have all runtime deps as pacman packages (best) or vendored wheels (worst). If you rely on pip, consider pre-installing to system site-packages during build to avoid network dependency at runtime.

Summary recommendation
Put build metadata + safe defaults in /etc/nyxus/.
Put UX things (.desktop, themes, configs) in /etc/skel or /usr/share.
Prefer pacman packages for fonts (e.g. ttf-inter) and other assets.
Avoid shipping /etc/shadow or secrets.
